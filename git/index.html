Git 撤回已 push 的注意事项

git revert 适合提交已经被推送到远程分支，适合公共分支，需要保留项目历史记录以便追踪问题。 如果其他人已经拉取了你要撤销的提交，请优先使用 git revert。 git reset 会重写提交历史，特别是在使用 --hard 或 --force 参数时。如果你执行了 git push --force，可能会覆盖远程分支的历史，影响其他协作者。不适合用于已经推送到远程仓库并被他人拉取的提交。该命令适用于提交仅存在于本地，尚未推送到远程仓库的情况。 在进行本地调试或清理提交历史时（如 rebase 后）可以使用。 在执行任何撤销操作前，建议先创建一个新分支或备份当前状态。 创建临时分支进行测试。

Git reset四种模式对比

模式对比表 模式（Mode） 工作区（Working Directory） 暂存区（Staging Area / Index） 本地版本库（HEAD） 说明 --soft 不修改 不修改 回退 只移动 HEAD 指针，不改变暂存区和工作区。适合撤销提交但保留所有更改为已暂存状态。 --mixed（默认） 不修改 回退 回退 移动 HEAD 并重置暂存区，但不改变工作区。这是默认行为，适合撤销提交并取消暂存更改，但保留文件内容。 --hard 回退（仅已追踪文件） 回退 回退 完全丢弃更改，将 HEAD、暂存区和工作区都恢复到指定提交的状态。危险操作，会丢失未提交的修改！ --keep 保留 Untracked 文件，回退已追踪文件 回退 回退 类似于 --hard，但不会删除工作区中未被追踪的文件（Untracked）。适用于想保留新添加的文件但丢弃旧更改。 说明 工作区（Working Directory） 当前你在文件系统中看到的文件状态，也就是你正在编辑的文件。 暂存区（Staging Area / Index） 准备下一次提交的文件快照，通过 git add 添加到暂存区的更改才会被下次提交包含。 本地版本库（Local Repository / HEAD） Git 存储的所有提交历史记录，HEAD 指向当前所在的提交。 使用场景建议 1. git reset --soft HEAD~1 使用场景：撤销最近的一次提交，但保留所有更改为“已暂存”状态。 结果：HEAD 被移动，但工作区和暂存区不变。 适用：提交后发现信息写错了，但不想重新添加文件。 2. git reset --mixed HEAD~1 或 git reset HEAD~1 使用场景：撤销提交并将更改从暂存区移除（但保留在工作区中）。 结果：HEAD 和暂存区被重置，工作区保持不变。 注意：这是默认行为，无需加 --mixed。 3. git reset --hard HEAD~1 使用场景：彻底丢弃最近一次提交的所有更改。 结果：HEAD、暂存区和工作区都被重置到上一个提交的状态。 ⚠️ 警告：所有未提交的更改都会被永久删除！ 4. git reset --keep HEAD~1 使用场景：想丢弃最近一次提交的更改，但保留工作区中新增的未追踪文件。 结果：HEAD 和暂存区被重置，已追踪文件被还原，但未追踪文件保留。 总结 --soft：只改 HEAD，什么都不动。 --mixed：改 HEAD 和暂存区，工作区不动。 --hard：全部重置，包括工作区（危险！） --keep：类似 hard，但不删未追踪文件。

Git中cherry-pick用法

介绍 目的是将 features (d e提交)分支中的某个特定提交（比如提交 d）合并到 master 分支，而另一个提交（比如提交 e）不合并。 操作步骤总结： 执行了以下 Git 命令： git checkout master # 切换到 master 分支 git cherry-pick <commit-d> # 将 features 分支中的提交 d 应用到当前分支 结果说明： commit-d 的更改内容被"复制"到了 master 分支上，作为一个新的提交（SHA 值不同）。 commit-e 不会被合并到 master，除非你再次执行 cherry-pick 或 merge。 features 分支本身没有改变，仍然包含 d 和 e 提交。
