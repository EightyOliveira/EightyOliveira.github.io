JVM中Synchronized实现机制

1 实现原理 每个Java对象都关联一个监视器锁（Monitor）。当线程进入synchronized代码块时，必须先获取该对象的Monitor，执行完后释放。 Monitor结构：在HotSpot虚拟机中，Monitor由C++实现，主要包含： _owner：指向当前持有锁的线程。 _WaitSet：存放调用wait()方法后被阻塞的线程。 _EntryList：存放等待获取锁的线程。 2 锁升级机制 无锁状态：对象刚创建时，没有线程竞争。 偏向锁：适用于只有一个线程访问同步块的场景。线程第一次进入时，JVM将对象头的Mark Word设置为偏向该线程ID。后续该线程进入无需再同步，直接执行。 轻量级锁：当有第二个线程竞争时，偏向锁升级为轻量级锁。线程通过CAS操作将Mark Word复制到自己的栈帧中，并尝试替换对象头为指向锁记录的指针。成功则获取锁，失败则自旋等待。 重量级锁：当竞争加剧（自旋超过一定次数），轻量级锁升级为重量级锁。此时未获取锁的线程进入阻塞状态，由操作系统调度，开销大但避免CPU空转。 3 字节码层面 synchronized代码块在编译后通过monitorenter和monitorexit指令实现： monitorenter：每个线程进入同步块时执行，尝试获取对象Monitor。 monitorexit：每个线程退出同步块时执行，释放Monitor。即使发生异常，也会通过异常表确保执行。

Java ReentrantLock

1. 实现 ReentrantLock 内部通过继承 AQS（AbstractQueuedSynchronizer）实现同步控制。 维护一个抽象内部类 Sync，继承自 AQS。 Sync 有两个实现： NonfairSync：非公平锁（默认） FairSync：公平锁 类关系： ReentrantLock Sync (extends AQS) NonfairSync FairSync 2. 加锁源码 final void lock() { if (compareAndSetState(0, 1)) setExclusiveOwnerThread(Thread.currentThread()); else acquire(1); } 步骤： 先尝试用 CAS 将 state 从 0 改为 1。 成功则设置当前线程为独占线程，获取锁。 失败则进入 acquire(1) 流程。 3. CAS compareAndSetState 基于 Unsafe 类的 CAS 操作： unsafe.compareAndSwapInt(this, stateOffset, expect, update); 参数说明（对应你提到的四个值）： this：Sync 实例（AQS 子类） stateOffset：state 字段的内存偏移量（如 16） expect：期望值（0） update：更新值（1） 作用：保证 state 的修改是原子的。 5. acquire(1) 流程 acquire 是 AQS 提供的方法： public final void acquire(int arg) { if (!tryAcquire(arg) && acquireQueued(addWaiter(Node.EXCLUSIVE), arg)) selfInterrupt(); } 步骤： tryAcquire：再次尝试获取锁（NonfairSync 中会尝试 CAS 或判断是否重入） addWaiter：构建 Node 节点，加入同步队列尾部 acquireQueued：在队列中自旋等待，尝试获取锁，失败则 park 阻塞 selfInterrupt：如果在等待过程中被中断，重新设置中断状态 6. state 状态管理 state 是 AQS 中的 volatile 变量，表示锁的持有状态。 state = 0：锁空闲 state >= 1：锁被占用 可重入性： 同一线程多次 lock，state 递增 每次 unlock，state 递减 state 回到 0 时彻底释放锁 7. 同步队列（AQS 队列） 当线程获取锁失败时，会被封装成 Node 节点加入 AQS 的双向链表队列。 队列中的线程通过 park/unpark 机制阻塞和唤醒。 公平锁：新线程必须排队，不能插队。 非公平锁：新线程可以先尝试抢锁（CAS），抢不到再排队。
